# 微信采集功能全新系统配置手册

适用场景：一台全新的 macOS 机器，准备启用“微信群采集 → 需求中心 → 人工审核”的稳定 OCR 采集流程。

当前稳定版本不接入 cc-connect，不需要机器人入群。采集方式是桌面微信可见窗口 OCR：系统会打开配置的微信群，截图当前可见聊天内容，OCR 识别需求信息，再推送到本机需求中心。

## 0. 先确认边界

微信采集功能会做：

- 搜索并打开你配置的微信群。
- 截取桌面微信窗口的可见聊天区域。
- OCR 识别文字和图片附件预览。
- 把识别出的需求、变更和附件摘要推送到需求中心。
- 定时采集时按每个群的采集间隔运行。

微信采集功能不会做：

- 不发送微信群消息。
- 不自动下载微信群文件。
- 不读取或破解微信聊天数据库。
- 不采集未配置的微信群。
- 不把 LLM 候选结果自动覆盖正式需求。

## 1. 准备系统环境

在配置页面前，先完成这些基础条件。

### 1.1 安装并登录桌面微信

- 安装 macOS 桌面微信。
- 用实际参与项目群的微信账号登录。
- 确认该账号本身已经在需要采集的微信群里。
- 确认微信群名称和页面里要填写的名称完全一致。

### 1.2 保持微信窗口可见

首次采集和定时采集都依赖桌面微信窗口：

- 微信必须正在运行。
- Mac 不能锁屏。
- 微信窗口不能被最小化。
- 首次历史补采前，最好把微信窗口停在一段包含有效需求信息的位置。

如果首次没有采到有效需求，系统不会推进 checkpoint；调整微信窗口位置后可以重新点“立即采集本群”。

### 1.3 授权 macOS 权限

微信 OCR 采集通常需要以下权限：

- 辅助功能：允许当前运行服务的终端、Codex 或相关运行器控制微信窗口。
- 屏幕录制：允许当前运行服务的终端、Codex 或相关运行器截图微信窗口。

如果权限不完整，后续“检查本机采集环境”或“立即采集本群”会失败；失败时先补权限，再重试，不需要重新配置群。

## 2. 启动本机服务

默认本机服务地址是：

```text
http://127.0.0.1:8765
```

在项目目录启动服务：

下面命令以当前项目路径 `/Users/ata/Documents/easy-exam-automation` 为例；如果全新机器上的项目路径不同，把项目路径替换成实际路径。

```bash
cd /Users/ata/Documents/easy-exam-automation
PORT=8765 CODEX_PYTHON=/Users/ata/.cache/codex-runtimes/codex-primary-runtime/dependencies/python/bin/python3 \
/Users/ata/.cache/codex-runtimes/codex-primary-runtime/dependencies/node/bin/node server/easy_exam_server.mjs
```

启动成功后打开：

```text
http://127.0.0.1:8765/wechat-collector
```

也可以打开系统配置页：

```text
http://127.0.0.1:8765/system-config
```

成功标志：

- 浏览器能打开页面。
- “微信采集”页面能正常加载群配置区域。
- “系统配置 → 微信采集配置”能看到“微信采集上线状态”。

## 3. 在“微信采集”页面配置微信群

进入：

```text
左侧导航 → 微信采集
```

### 3.1 新增群

点击“新增群”，填写：

- 微信群：必须和桌面微信里的群名称一致。
- 项目名称：用于需求中心展示和后续执行识别。
- 客户名称：用于需求单客户信息。
- 关联需求单编号：可留空。
- 启用：需要采集就选“是”。
- 采集间隔：建议先用 `15` 分钟，必须是大于等于 1 的整数。
- 历史补采方式：默认使用 `OCR 当前窗口`。

关联需求单编号的含义：

- 留空：首次有效采集时新建需求单。
- 填写：后续采集和变更都归到这张需求单。

### 3.2 保存本群设置

展开群卡片后，在“群设置操作”里点击“保存本群设置”。

成功标志：

- 页面提示本群已保存。
- 群卡片里能看到项目、客户、需求编号或 `--`。
- 群配置不会因为刷新页面丢失。

注意：

- 每个群单独保存，便于分群管理。
- 群名重复会被拒绝。
- 采集间隔不是正整数会被拒绝。

## 4. 做首次历史补采

首次历史补采的目标是：把已有微信群需求内容先采进需求中心，并建立后续去重用的 checkpoint。

操作顺序：

1. 在桌面微信里打开目标群。
2. 把聊天窗口停在包含有效需求信息的位置。
3. 回到“微信采集”页面。
4. 在对应群卡片点击“立即采集本群”。
5. 确认弹窗后等待运行完成。

成功标志：

- 群卡片状态显示成功或已推送。
- 群卡片里出现需求编号。
- “需求更新记录”里出现采集运行记录。
- “需求单记录”页面能看到对应需求。

如果首次没有采到有效需求：

- 状态可能是 `no_requirement_signal`。
- 这不是成功的首次补采。
- 系统不会推进 checkpoint。
- 需要把微信窗口调整到包含需求的位置，再点“立即采集本群”。

如果提示没有新消息：

- 状态可能是 `no_new_messages`。
- 表示当前可见内容和已处理内容重复。
- 如果你想补采更早的内容，需要把微信滚动到未处理的历史需求位置再采。

## 5. 在“系统配置”页面做上线前检查

进入：

```text
左侧导航 → 系统配置 → 微信采集配置
```

先看“微信采集上线状态”，重点确认：

- 本机服务：正常。
- OCR：正常。
- 需求中心：正常。
- 群验证：每个启用群都通过。
- 链路自检：有效。
- 自动采集：上线后应显示已安装或已加载。

### 5.1 检查本机采集环境

点击“检查本机采集环境”。

这个动作会检查：

- 微信窗口是否可读。
- OCR helper 是否可用。
- 截图区域是否合理。
- 群配置是否合法。

这个动作不会：

- 不激活微信。
- 不切换群。
- 不截图。
- 不 OCR。
- 不推送需求中心。

成功标志：

- “最近一次预检”显示通过。
- 如果失败，按错误提示修复微信窗口、权限或群配置后重试。

### 5.2 运行链路自检

点击“运行链路自检”。

这个动作会用临时数据验证：

- 需求能写入需求中心。
- 变更能进入人工审核。

这个动作不会操作真实微信，也不会污染正式需求库。

成功标志：

- “链路自检结果”显示通过。
- 通过结果 24 小时内有效。

### 5.3 逐群真实验证

回到“微信采集”页面，对每个启用群执行一次“立即采集本群”。

要求：

- 每个启用群都要做。
- 每个群最近 24 小时内都要有真实微信成功推送记录。
- 没有采到有效需求的群，不算通过上线门槛。

## 6. 可选：配置 LLM 候选解析

进入：

```text
系统配置 → 微信采集配置 → LLM 候选解析
```

默认不开启 LLM，系统使用本地规则解析微信群需求。

如果开启 LLM：

1. 启用 LLM：选“是”。
2. 服务商：选择 OpenAI 或千问。
3. Endpoint：一般不用改，系统会带出默认地址。
4. API Key：填写密钥。
5. 点击“验证并读取模型”。
6. 从读取到的模型列表中选择模型。
7. 点击“保存 LLM 配置”。

成功标志：

- 页面显示 API Key 已配置。
- 模型列表能读取成功。
- 后续采集结果会在需求详情的“解析结果确认”里展示候选结构化结果和原文证据。

注意：

- LLM 只生成候选结果。
- 候选结果不会自动覆盖正式需求。
- LLM 失败时，规则解析仍会继续。

## 7. 安装整套自动采集

进入：

```text
系统配置 → 微信采集配置 → 2. 上线自动采集
```

只有上线门槛全部通过后，“安装整套自动采集”才应该可用。

### 7.1 先部署 launchd 运行副本

正式自动采集由 macOS LaunchAgent 执行。LaunchAgent 不直接运行仓库目录，而是运行：

```text
~/Library/Application Support/easy-exam-automation/app
```

因此，全新系统第一次安装自动采集前，先执行：

```bash
cd /Users/ata/Documents/easy-exam-automation
/Users/ata/.cache/codex-runtimes/codex-primary-runtime/dependencies/node/bin/node \
  scripts/deploy_launchd_runtime.mjs --migrate-runtime
```

成功标志：

- 输出 JSON 里 `ok` 为 `true`。
- `appDir` 指向 `~/Library/Application Support/easy-exam-automation/app`。
- `runtimeDir` 指向 `~/Library/Application Support/easy-exam-automation/runtime`。

后续只是更新代码时，可以再次运行同一脚本但不带 `--migrate-runtime`；这样会刷新 app 副本，不覆盖已有运行数据。

注意：当前 launchd 模板按本机用户目录生成。如果全新系统使用的 macOS 用户不是 `ata`，需要先确认 `deploy/*.plist.template` 里的用户路径已经改成实际用户目录，再安装自动采集。

### 7.2 安装自动采集

上线门槛包括：

- 微信群配置合法。
- 至少一个微信群已启用。
- 本机服务可达。
- OCR 可用。
- 链路自检 24 小时内通过。
- 每个启用群 24 小时内真实采集并成功推送。

点击“安装整套自动采集”后，系统会：

1. 安装本机 easy-exam 服务。
2. 安装微信群采集定时任务。
3. 按群配置的采集间隔定时运行。

成功标志：

- “微信采集上线状态”里自动采集显示正常。
- “高级维护：服务与定时任务”里本机服务和定时任务显示已加载。
- 后续“微信采集”页面的运行历史持续出现新记录。

如果安装失败：

- 页面会显示缺少哪个门槛。
- API 会拒绝半安装。
- 如果服务已装但定时安装失败，系统会尝试回滚服务安装。

## 8. 日常使用流程

日常主要看两个页面。

### 8.1 微信采集

用于：

- 新增或停用微信群。
- 修改项目、客户、关联需求单编号。
- 立即采集本群。
- 查看本群任务执行状态。
- 查看本群需求更新记录。
- 查看本群附件关联状态。

日常判断：

- 群卡片状态正常：说明最近采集没有明显异常。
- 需求更新记录有新内容：说明采集进入需求中心。
- 附件状态为已关联：说明当前可见聊天里提到的文件名匹配到了本机已下载附件。

### 8.2 需求单记录

用于处理采集后的需求。

建议按“需求处理工作台”的下一步动作处理：

- 有待处理变更：先处理变更。
- 需客户补充：生成客户补充话术。
- 待客户确认：确认客户是否已回复。
- 可进入执行：创建考试任务并记录任务编号。

不要只按需求编号找需求；优先按下一步动作处理。

## 9. 附件处理规则

系统只处理本机已下载的微信附件。

附件进入需求的条件：

1. 文件已经存在于本机微信下载目录。
2. 当前群可见聊天内容里出现了这个文件名。
3. 本轮采集能把文件名和本群消息关联上。

如果文件后来才人工下载：

1. 先在微信里把包含该文件消息的聊天位置露出来。
2. 再点“立即采集本群”。
3. 查看群卡片里的“查看附件状态”。

当前附件正文预览能力：

- `.txt`、`.csv`：读取文本预览，支持 UTF-8，失败时用 GB18030 兜底。
- `.xlsx`：读取工作表和单元格文本预览。
- `.png`、`.jpg`、`.jpeg`：使用 macOS Vision OCR 读取图片文字。
- `.xls`、`.pdf`、`.docx`、`.doc`：当前只记录文件名、类型、大小、修改时间，不读取正文内容。

## 10. 常见问题处理

### 页面打不开

检查本机服务是否启动：

```text
http://127.0.0.1:8765/api/health
```

如果不可达，重新启动本机服务。

### “检查本机采集环境”失败

优先检查：

- 微信是否运行。
- 微信窗口是否可见。
- macOS 辅助功能权限是否授权。
- macOS 屏幕录制权限是否授权。
- 群名是否和微信里完全一致。

### “立即采集本群”没有采到需求

判断状态：

- `no_requirement_signal`：可见内容没有识别到需求字段或变更；调整微信窗口到有效需求位置后重试。
- `no_new_messages`：当前内容已处理过；滚动到未处理内容后重试。
- `failed`：通常是微信窗口、权限、截图或需求中心可达性问题；先看错误信息。

### 自动采集按钮不可用

通常是上线门槛没过：

- 链路自检超过 24 小时。
- 某个启用群没有最近 24 小时真实采集成功推送。
- 本机服务不可达。
- OCR 不可用。
- 群配置不合法。

按“系统配置 → 微信采集上线状态”的待处理项逐个修复。

### 定时采集长时间没有新记录

检查：

- Mac 是否睡眠或锁屏。
- 微信是否退出登录。
- 本机服务是否仍可达。
- 定时任务是否仍加载。
- “运行日志”里是否有最近错误。

### 附件没有进入需求

常见原因：

- 文件还没有被微信下载到本机。
- 当前可见聊天里没有出现文件名。
- 文件名和本机文件名不一致。
- 文件属于另一个群，当前群消息没有提到它。

处理方式：

- 先人工下载附件。
- 把微信窗口停在包含该文件消息的位置。
- 重新执行“立即采集本群”。

## 11. 推荐上线顺序清单

按下面顺序做，不要跳步：

1. 登录桌面微信，并确认账号在目标群里。
2. 启动本机服务，打开 `/wechat-collector`。
3. 在“微信采集”新增微信群。
4. 保存每个群的本群设置。
5. 在“系统配置”执行“检查本机采集环境”。
6. 在“系统配置”执行“运行链路自检”。
7. 回到“微信采集”，逐个群执行“立即采集本群”。
8. 到“需求单记录”确认需求已进入处理工作台。
9. 如果需要 LLM，配置并验证模型后保存。
10. 回到“系统配置”，确认“微信采集上线状态”全部正常。
11. 点击“安装整套自动采集”。
12. 观察一次定时运行历史，确认自动采集持续产出记录。
